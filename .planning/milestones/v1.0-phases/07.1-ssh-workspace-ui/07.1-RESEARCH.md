# Phase 7.1: SSH Workspace UI - Research

**Researched:** 2026-03-26
**Domain:** GTK4 dialog UI, SSH config parsing, TOML host persistence
**Confidence:** HIGH

## Summary

This phase adds a keyboard-driven UI for creating SSH workspaces. The entire backend (SSH lifecycle, bridge, tunnel, remote daemon deployment) already exists from Phase 7. The work is purely UI: a GTK4 dialog triggered by Ctrl+Shift+S, with text entry for `user@host`, autocomplete from saved hosts and `~/.ssh/config`, and auto-save of successful connections.

The codebase has clear patterns for all required pieces: `ShortcutAction` enum + `ShortcutMap` for keybindings, `AlertDialog` for modal dialogs, `gtk4::Entry` for text input, and `create_remote_workspace()` + `run_ssh_lifecycle()` for the SSH backend. The main new work is: (1) an SSH connect dialog with autocomplete, (2) a hosts file manager, and (3) a simple SSH config parser for Host aliases.

**Primary recommendation:** Build a custom `gtk4::Window` (modal, transient) with a `gtk4::Entry` using `gtk4::EntryCompletion` for autocomplete. Parse `~/.ssh/config` Host lines with a simple regex -- no external crate needed. Store saved hosts in `~/.config/cmux/ssh_hosts.toml` as a flat TOML array.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Simple text entry dialog -- single input field for `user@host` (or `user@host:port`). Like a browser address bar for SSH. No separate form fields.
- **D-02:** Dropdown autocomplete showing matching hosts from saved hosts file and `~/.ssh/config` as user types. Standard GTK4 completion pattern.
- **D-03:** Ctrl+Shift+S triggers the SSH connect dialog. Add `NewSshWorkspace` variant to `ShortcutAction` enum. Configurable via `config.toml` like all other shortcuts.
- **D-04:** SSH hosts saved to separate file at `~/.config/cmux/ssh_hosts.toml` (not in main config.toml). Hosts auto-saved after successful connection.
- **D-05:** Parse `~/.ssh/config` Host entries for autocomplete suggestions in the dialog. The SSH command already uses the config transparently -- this just surfaces aliases in the UI. No need to parse identity files, ports, or proxy settings.
- **D-06:** Sidebar shows connection state (connecting/failed/connected) via existing `ConnectionState` enum. Terminal pane shows error text like `[SSH: Connection refused -- user@host:22]`. Reuses existing Phase 7 patterns.
- **D-07:** Rely on ssh-agent and system SSH command. No in-app password prompts. If auth fails, error appears in terminal pane.
- **D-08:** Auto-name from hostname (extract from `user@host`, use just the host part). User can rename later with Ctrl+Shift+R.

- Whether to deduplicate between ssh_hosts.toml and ~/.ssh/config entries

### Pattern 1: Shortcut -> Handler -> Dialog -> Backend
**What:** Same pattern as all other shortcuts: enum variant -> match arm -> handler function -> action
**When to use:** Always for keyboard-triggered actions

The flow:
1. `config.rs`: Add `NewSshWorkspace` variant to `ShortcutAction`, `new_ssh_workspace: Option<String>` to `ShortcutConfig`, default binding `<Ctrl><Shift>s`
2. `config.rs`: Add to `KNOWN_SHORTCUTS` array and `ShortcutMap::from_config` entries table
3. `shortcuts.rs`: Add match arm `Some(ShortcutAction::NewSshWorkspace) => { handle_new_ssh_workspace(...); Propagation::Stop }`
4. `ssh_dialog.rs`: `handle_new_ssh_workspace` opens the dialog

### Pattern 2: Modal Dialog with Entry + EntryCompletion
**What:** Custom `gtk4::Window` configured as modal + transient-for parent
**When to use:** Any dialog needing custom input widgets beyond buttons

```rust
// Source: existing handle_close_workspace pattern + GTK4 docs
pub fn show_ssh_dialog(
    app: &gtk4::Application,
    state: Rc<RefCell<AppState>>,
) {
    let window = app.windows().into_iter().next();

    let dialog = gtk4::Window::builder()
        .title("Connect to SSH Host")
        .modal(true)
        .default_width(400)
        .default_height(80)
        .build();
    if let Some(parent) = window.as_ref() {
        dialog.set_transient_for(Some(parent));
    }

    let entry = gtk4::Entry::new();
    entry.set_placeholder_text(Some("user@host or host alias"));

    // Set up EntryCompletion with host list
    let completion = gtk4::EntryCompletion::new();
    let store = gtk4::ListStore::new(&[glib::Type::STRING]);
    // ... populate store from ssh_hosts + ssh_config ...
    completion.set_model(Some(&store));
    completion.set_text_column(0);
    completion.set_minimum_key_length(1);
    entry.set_completion(Some(&completion));

    // Enter key -> create SSH workspace
    entry.connect_activate({
        let state = state.clone();
        let dialog = dialog.clone();
        move |e| {
            let target = e.text().to_string().trim().to_string();
            if !target.is_empty() {
                trigger_ssh_connect(&state, target);
                dialog.close();
            }
        }
    });

    // Escape -> close dialog
    let esc_ctrl = gtk4::EventControllerKey::new();
    esc_ctrl.connect_key_pressed({
        let dialog = dialog.clone();
        move |_, key, _, _| {
            if key == gtk4::gdk::Key::Escape {
                dialog.close();
                gtk4::glib::Propagation::Stop
            } else {
                gtk4::glib::Propagation::Proceed
            }
        }
    });
    dialog.add_controller(esc_ctrl);

    dialog.set_child(Some(&entry));
    dialog.present();
    entry.grab_focus();
}
```

### Pattern 3: SSH Workspace Creation from Dialog
**What:** Reuse the exact same bridge + lifecycle spawn pattern from socket handler
**When to use:** When creating SSH workspace from UI (vs socket command)

```rust
// Source: src/socket/handlers.rs lines 88-111 (WorkspaceCreate with remote_target)
fn trigger_ssh_connect(state: &Rc<RefCell<AppState>>, target: String) {
    let (write_tx, write_rx) = tokio::sync::mpsc::unbounded_channel();
    let (output_tx, _output_rx) = tokio::sync::mpsc::unbounded_channel();
    let bridge = std::sync::Arc::new(SshBridge::new(write_tx, write_rx, output_tx));
    let id = state.borrow_mut().create_remote_workspace(target.clone(), &bridge);
    state.borrow_mut().workspace_bridges.insert(id, bridge.clone());

    let ssh_tx = state.borrow().ssh_event_tx.clone();
    let rt_handle = state.borrow().runtime_handle.clone();
    if let (Some(tx), Some(rt)) = (ssh_tx, rt_handle) {
        let handle = rt.spawn(run_ssh_lifecycle(id, target, tx, bridge));
        state.borrow_mut().ssh_task_handles.insert(id, handle);
    }
}
```

### Pattern 4: SSH Config Parsing (Host aliases only)
**What:** Read `~/.ssh/config` line by line, extract `Host` directive values
**When to use:** For autocomplete suggestions

```rust
// Simple parser -- no crate needed
fn parse_ssh_config_hosts() -> Vec<String> {
    let path = dirs_or_home().join(".ssh/config");
    let content = match std::fs::read_to_string(&path) {
        Ok(s) => s,
        Err(_) => return Vec::new(),
    };
    let mut hosts = Vec::new();
    for line in content.lines() {
        let trimmed = line.trim();
        if let Some(rest) = trimmed.strip_prefix("Host ").or_else(|| trimmed.strip_prefix("Host\t")) {
            for host in rest.split_whitespace() {
                // Skip wildcard patterns
                if !host.contains('*') && !host.contains('?') {
                    hosts.push(host.to_string());
                }
            }
        }
    }
    hosts
}
```

### Pattern 6: Auto-Save on Successful Connection
**What:** When SSH `ConnectionState::Connected` event arrives, save host to file
**When to use:** In the existing SSH event processing timer (100ms GTK timer)

The existing `SshEvent::StateChanged` handler in `main.rs` already processes connection state changes. Add a check: when state transitions to `Connected`, look up the workspace's `remote_target` and append it to `ssh_hosts.toml` if not already present.

### Anti-Patterns to Avoid
- **Don't create a new async runtime for the dialog.** The tokio runtime handle is already on AppState. Use it.
- **Don't duplicate the SSH bridge/lifecycle spawn code.** Extract the common pattern from `socket/handlers.rs` into a shared helper function, then call from both socket handler and dialog handler.
- **Don't over-parse SSH config.** Only Host aliases are needed. Parsing HostName, IdentityFile, ProxyJump etc is out of scope (D-05).
- **Don't block the GTK main thread** for file I/O. SSH config and hosts file are tiny (< 1KB typically), so synchronous reads are fine in the dialog-open path.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Text entry autocomplete | Custom popup/dropdown widget | `gtk4::EntryCompletion` | Works in GTK 4.12, matches the "standard GTK4 completion pattern" from D-02 |
| TOML serialization | Manual string formatting | `toml::to_string_pretty` + `toml::from_str` | Already in deps, handles escaping |
| SSH config parsing | Full parser with all directives | Simple line-by-line Host extraction | 20 lines vs adding a crate; we only need host aliases |

## Common Pitfalls

### Pitfall 1: EntryCompletion Deprecation Warnings
**What goes wrong:** Compiler warnings about deprecated API
**Why it happens:** `gtk4::EntryCompletion` is deprecated since GTK 4.10
**How to avoid:** Add `#[allow(deprecated)]` on the module or function that uses it. There is no replacement -- GTK maintainers confirm it should still be used.
**Warning signs:** CI lint failures from deprecation warnings

### Pitfall 2: Dialog Focus Stealing After Close
**What goes wrong:** After dialog closes, keyboard input goes nowhere (not back to terminal)
**Why it happens:** GTK4 modal windows don't always restore focus correctly
**How to avoid:** After `dialog.close()`, explicitly call `focus_active_surface()` on the active split engine to restore Ghostty focus
**Warning signs:** Typing after Escape/Enter from dialog does nothing

### Pitfall 3: Duplicate SSH Workspace Creation Code
**What goes wrong:** Socket handler and dialog handler diverge over time
**Why it happens:** Copy-pasting the bridge+lifecycle spawn pattern
**How to avoid:** Extract into a shared helper on AppState (e.g., `AppState::spawn_ssh_workspace(target)`)
**Warning signs:** Bug fixes applied to one path but not the other

### Pitfall 4: Race Between Dialog Submit and Connection State
**What goes wrong:** User submits dialog, workspace appears, but connection state update arrives before sidebar row is fully built
**Why it happens:** SSH lifecycle spawns on tokio, immediately sends `Reconnecting(0)` state event
**How to avoid:** `create_remote_workspace` already initializes with `Reconnecting(0)` state -- the first StateChanged event is a no-op update. No race.

### Pitfall 5: SSH Config with Multiple Host Entries Per Line
**What goes wrong:** Parser only captures first host alias per line
**Why it happens:** `Host bastion jump-host dev-server` is valid SSH config (multiple aliases on one line)
**How to avoid:** Split on whitespace after `Host ` prefix, iterate all tokens
**Warning signs:** Missing hosts from SSH config in autocomplete

### Pitfall 6: Hosts File Write Without Directory Creation
**What goes wrong:** First save fails because `~/.config/cmux/` doesn't exist
**Why it happens:** Fresh install, no config.toml created yet
**How to avoid:** `std::fs::create_dir_all` before writing ssh_hosts.toml
**Warning signs:** Silent save failure on first SSH connection

### Workspace Auto-Naming from Host (D-08)
```rust
fn workspace_name_from_target(target: &str) -> String {
    // "user@host.example.com" -> "host.example.com"
    // "host.example.com" -> "host.example.com"
    // "dev-server" -> "dev-server" (SSH alias)
    let host_part = target.split('@').last().unwrap_or(target);
    // Strip port if present: "host:2222" -> "host"
    let without_port = host_part.split(':').next().unwrap_or(host_part);
    without_port.to_string()
}
```

Note: `Workspace::new_remote` currently uses `format!("SSH: {}", target)`. The name should instead use the extracted hostname per D-08. Either modify `new_remote` or rename after creation.
