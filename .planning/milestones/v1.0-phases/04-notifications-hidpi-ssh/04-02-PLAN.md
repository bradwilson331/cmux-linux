**1. Audit existing scale-factor handler in surface.rs:**

Read the `notify::scale-factor` signal handler on GLArea (around line 368). Verify it:
- Calls `ghostty_surface_set_content_scale` with the new scale factor as f64
- Calls `ghostty_surface_refresh` or `queue_render` to trigger a redraw
- Passes the scale factor correctly (GTK4 `Widget::scale_factor()` returns i32 for integer scales; for fractional scaling on Wayland, GTK4 4.12+ reports the actual fractional value through the native Wayland protocol but `scale_factor()` still returns the ceiling integer -- verify whether `ghostty_surface_set_content_scale` needs the exact fractional value or the integer ceiling)

Per D-10: Rely on GTK4's native Wayland fractional scaling. No custom fractional logic.

**2. If the handler passes integer scale_factor() only:**

Check if there's a way to get the actual fractional scale from GTK4. In GTK4 4.14+, the surface's `gdk::Surface::scale()` returns a f64 fractional value. If the current handler uses `widget.scale_factor()` (integer), consider updating to:
```rust
// Get fractional scale if available
let scale = gl_area.native()
    .and_then(|n| n.surface())
    .map(|s| s.scale())
    .unwrap_or(gl_area.scale_factor() as f64);
```

Only make this change if `gdk::Surface::scale()` is available in gtk4-rs 0.10. If not available in this version, document the limitation and keep the integer scale.

**3. Verify CSS doesn't have HiDPI issues:**

Review APP_CSS for any values that might render incorrectly at fractional scales:
- `min-width: 4px` on separators -- GTK4 handles subpixel rendering natively, this is fine
- `min-height: 36px` on rows -- fine, GTK4 scales these
- `border: 1px solid` on active-pane -- verify this doesn't become invisible at 2x (may need to stay as-is since GTK4 handles border rendering)
- `font-size: 14px` -- GTK4 handles font scaling natively

If the attention-dot CSS from Plan 01 uses `min-width: 8px`, verify it renders as a visible dot at all scales. At 2x it becomes 16 device pixels (visible). At 0.5x it would be 4 device pixels (still visible). No change expected.

**4. Add a scale-factor change log line for debugging:**

In the scale-factor handler, add an `eprintln!` to log scale changes:
```rust
eprintln!("cmux: scale-factor changed to {} for surface", scale);
```
This helps diagnose HiDPI issues in the field.

**5. Verify that the initial surface creation passes the correct scale:**

In `create_surface()`, check that the initial `ghostty_surface_set_content_scale` call (if any) uses the GLArea's scale factor at realize time, not a hardcoded 1.0. If the initial scale is set in the realize callback, verify it reads from the widget.
