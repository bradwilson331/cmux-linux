# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Workspaces + Pane Splits.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** This is a GTK4 native Linux desktop application written in Rust.
> There is no web stack, no shadcn, no Tailwind CSS. All design tokens below map to
> GTK4 CSS provider rules applied via `gtk_css_provider_load_from_data()`, and to
> GtkWidget property values set in Rust code. "px" means logical pixels at 1x scale;
> GTK4 scales these automatically via `scale_factor()` for HiDPI.

## Color

GTK4 CSS applied via a custom `GtkCssProvider` loaded at app startup. Use these exact hex values as the Phase 2 baseline. These are dark-theme values matching Ghostty's default dark terminal background. A light theme is out of scope for Phase 2.

| Role | Hex | GTK4 CSS Variable | Usage |
|------|-----|-------------------|---------|
| Dominant (60%) | `#1a1a1a` | `@dominant` | Main window background, terminal area background (behind GtkGLArea) |
| Secondary (30%) | `#242424` | `@secondary` | Sidebar background (`GtkListBox` background), `GtkPaned` background |
| Accent (10%) | `#5b8dd9` | `@accent` | Active workspace row highlight only; active pane divider indicator |

Accent reserved for:
1. Active workspace row background in the `GtkListBox` sidebar

## Component Inventory

Phase 2 introduces these GTK4 widget compositions. The executor must implement each exactly as described.

### 1. Sidebar (`src/sidebar.rs`)

```
GtkScrolledWindow
  └── GtkListBox (selection-mode = single)
        └── GtkListBoxRow (one per workspace)
              └── GtkLabel (text = workspace name)
```

- Width: 160px fixed (set via `set_size_request(160, -1)` on the `GtkScrolledWindow`)
- Background: `@secondary` (#242424)
- Row height: 36px (`min-height: 36px` in CSS)
- Row padding: 8px top/bottom, 16px left/right
- Active row: background `@accent` (#5b8dd9), text color `#ffffff`, font-weight 600
- Inactive row: background transparent, text color `#cccccc`, font-weight 400
- Hover row (not active): background `#2e2e2e`
- Scrollbar: GTK4 overlay scrollbar (default behavior — do not override)
- No icons in Phase 2 (CONTEXT.md D-02)

### 2. Workspace/Terminal Area

```
GtkBox (orientation = horizontal)
  ├── Sidebar (GtkScrolledWindow, see above)
  └── Terminal content area (GtkStack or direct GtkWidget swap)
        └── Per-workspace root: GtkPaned or GtkGLArea (leaf)
```

- The terminal area fills all remaining width after sidebar
- Background behind GLArea widgets: `@dominant` (#1a1a1a)

### 3. Pane Split Tree (`src/split_engine.rs`)

```
SplitNode::Leaf  → GtkGLArea (Ghostty surface)
SplitNode::Split → GtkPaned
                     ├── left/top child (SplitNode)
                     └── right/bottom child (SplitNode)
```

- `GtkPaned` orientation: `Horizontal` for split-right (`Ctrl+D`), `Vertical` for split-down (`Ctrl+Shift+D`)
- Initial position: 50% of available space (call `paned.set_position(available / 2)` after realize)
- Divider style: GTK4 default separator (do not override handle appearance in Phase 2)
- Active pane indicator: 1px solid `@accent` border on the currently focused `GtkGLArea` widget
  - CSS: `.active-pane { border: 1px solid #5b8dd9; }`
  - Applied by adding/removing the `active-pane` CSS class on focus change

### 4. Window Layout

```
ApplicationWindow (default_width=800, default_height=600)
  └── GtkBox (orientation = horizontal)
        ├── Sidebar GtkScrolledWindow (160px, shown/hidden via Ctrl+B)
        └── GtkStack (one page per workspace)
```

- Sidebar visibility: toggled by calling `.set_visible(bool)` on the sidebar widget (CONTEXT.md D-04)
- No animation on sidebar toggle in Phase 2 (GTK4 `GtkRevealer` animation deferred to later phase)

## Interaction Contract

### Focus Indicator

- Exactly one `GtkGLArea` is "active" at all times within the visible workspace
- Active pane: add CSS class `active-pane` → 1px `#5b8dd9` border
- Inactive panes: no border
- On split: new pane receives focus immediately
- On pane close: focus moves to the surviving sibling
- On workspace switch: focus moves to the last-focused pane within that workspace

### Divider Drag

- Handled natively by `GtkPaned` — no custom hit-testing or CSS override required
- Minimum pane size: GTK4 natural minimum from `GtkGLArea` (approximately 40x40px)
- No snap-to-half behavior (drag freely)

### Workspace Switch

- Click on sidebar row: switch immediately, no animation
- `Ctrl+1` through `Ctrl+9`: switch by 1-indexed position
- `Ctrl+]` / `Ctrl+[`: next/previous workspace, wrap-around at ends
- Switching does not change GtkPaned positions within the workspace

### Sidebar Toggle

- `Ctrl+B`: toggle sidebar visibility
- Terminal area expands/contracts to fill the freed space (natural GTK4 GtkBox behavior)
- No transition animation in Phase 2

### Rename Inline UI

- `Ctrl+Shift+R`: replace the `GtkLabel` in the active row with a `GtkEntry` (same dimensions)
- Press `Enter` or focus-out: commit the new name, replace `GtkEntry` with `GtkLabel`
- Press `Escape`: cancel rename, restore previous name
- No separate dialog or popover in Phase 2 (CONTEXT.md D-03)

## States Required

Each GTK4 widget must handle these states. Executor must implement all states before marking a component complete.

### Sidebar Row States
- [ ] Normal (inactive workspace, not hovered)
- [ ] Hover (inactive workspace, pointer over row)
- [ ] Active (current workspace)
- [ ] Active + rename in progress (GtkEntry visible)

### Pane States
- [ ] Normal (not focused)
- [ ] Active (focused, `active-pane` CSS class applied)

### Window States
- [ ] Sidebar visible
- [ ] Sidebar hidden (Ctrl+B toggled)
- [ ] Single pane (no split — no GtkPaned, just GtkGLArea)
- [ ] Split with 2 panes (one GtkPaned)
- [ ] Split with 3+ panes (nested GtkPaneds)

### Empty States
- [ ] Zero workspaces (app just cleared all — show empty state label before forcing a new workspace)
