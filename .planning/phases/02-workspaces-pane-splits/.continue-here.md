  (cursor_c_cancel in xev loop) — by the time set_focus(true) is processed, cursor timer

  See: ghostty/src/renderer/Thread.zig line 407: `if (self.cursor_c.state() != .active)`

<context>

The rendering pipeline in Ghostty embedded mode:
1. Renderer thread cursor timer fires → cursor_blink_visible flipped → wakeup.notify()
2. Renderer wakeupCallback → drainMailbox → renderCallback
3. renderCallback → updateFrame(t.state, t.flags.cursor_blink_visible) → drawFrame(false)
4. drawFrame(false) with must_draw_from_app_thread=true:
   IF hasVsync(): early return, NO redraw_surface pushed  ← SUSPECTED BUG PATH
   ELSE: push redraw_surface to app_mailbox
5. App processes redraw_surface via ghostty_app_tick → action_cb(.render) → queue_render
6. GTK render → ghostty_surface_draw → Surface.draw() → renderer.drawFrame(true)

cmux's wakeup_cb also calls queue_render directly (step 6 happens regardless of step 4-5).
But queue_render only presents the LAST frame baked in step 3's updateFrame call.
If hasVsync=true, step 3 runs but step 4 skips — the baked frame IS updated with new
cursor_blink_visible, but the app thread is never TOLD to redraw. The stale frame
from the PREVIOUS updateFrame call keeps being presented.
