# Phase 12.1: CLI Browser Parity & Skill Packaging - Research

**Researched:** 2026-03-29
**Domain:** Rust CLI restructuring (clap nested subcommands), FHS packaging, agent skill discovery
**Confidence:** HIGH

## Summary

This phase restructures the flat `BrowserOpen`/`BrowserClick`/etc. CLI commands into a nested `cmux browser <surface> <action>` subcommand hierarchy, adds surface handle management (short refs like `surface:N` mapped to UUIDs), and packages the `cmux` and `cmux-browser` skills into .deb/.rpm packages at `/usr/share/cmux/skills/`.

The codebase is well-positioned for this work. The existing flat browser commands (23 variants in `Commands` enum) already map to `browser.*` JSON-RPC methods through a consistent `command_to_rpc()` dispatch pattern. The socket-side handler already has a generic `BrowserAction` proxy that strips the `browser.` prefix and forwards to agent-browser. The packaging scripts from Phase 12 follow a clear `install -Dm` pattern that is straightforward to extend.

**Primary recommendation:** Implement in three streams: (1) CLI restructure with nested `Browser` subcommand enum, (2) surface handle registry in cmux-app with socket-side ref resolution, (3) skill/CLAUDE.md packaging additions to build scripts.

## Architecture Patterns

### CLI Restructure Pattern

The existing `Commands` enum has 23 flat `Browser*` variants. These need to collapse into a single `Browser` variant with a nested sub-enum.

```rust
#[derive(Subcommand)]
pub enum Commands {
    // ... existing non-browser commands unchanged ...

    /// Browser automation (agent primary interface)
    #[command(subcommand)]
    Browser(BrowserCommand),

    // Hidden backward-compat aliases for old flat syntax
    #[command(hide = true)]
    BrowserOpen { url: String },
    // ... other flat variants hidden ...
}

#[derive(Subcommand)]
pub enum BrowserCommand {
    /// Open a URL in a new browser surface
    Open {
        url: String,
        #[arg(long)]
        workspace: Option<String>,
    },
    /// List browser surfaces
    List,
    /// Target a specific surface for an action
    #[command(external_subcommand)]
    Surface(Vec<String>),
}
```

**Key insight for D-04 (surface targeting):** clap's `external_subcommand` pattern can capture `surface:N <action> [args...]` as a `Vec<String>`. However, a cleaner approach is to parse the first positional argument manually in the `run()` function. When the first token after `browser` matches `surface:\d+` or a UUID pattern, treat it as a surface target and parse the remaining args as the action. This avoids clap complexity and matches how the skills document the syntax.

**Recommended approach:** Use a dedicated parser in `run()` that intercepts the `Browser` variant, checks if the first subcommand token is a surface ref, and routes accordingly. The actual clap enum should be:

```rust
#[derive(Subcommand)]
pub enum BrowserCommand {
    /// Open a URL in a new browser surface
    Open {
        url: String,
        #[arg(long)]
        workspace: Option<String>,
        #[arg(long, default_value = "json")]
        format: String,
    },
    /// List browser surfaces
    List,
    /// Close a browser surface
    Close {
        /// Surface ref or UUID
        surface: Option<String>,
    },
    /// Take a snapshot of the page
    Snapshot {
        /// Surface ref or UUID (required)
        surface: String,
        #[arg(long)]
        interactive: bool,
        #[arg(long)]
        compact: bool,
        #[arg(long)]
        max_depth: Option<u32>,
    },
    /// Click an element
    Click {
        surface: String,
        /// Element ref (e1, e2...) or CSS selector
        target: String,
        #[arg(long)]
        snapshot_after: bool,
    },
    /// Fill an input field
    Fill {
        surface: String,
        target: String,
        text: String,
        #[arg(long)]
        snapshot_after: bool,
    },
    // ... etc for each action
    /// Wait for a condition
    Wait {
        surface: String,
        #[arg(long)]
        selector: Option<String>,
        #[arg(long)]
        text: Option<String>,
        #[arg(long)]
        url_contains: Option<String>,
        #[arg(long)]
        load_state: Option<String>,
        #[arg(long)]
        function: Option<String>,
        #[arg(long, default_value = "30000")]
        timeout_ms: u64,
    },
    /// Get a property (url, title, text, html, value, etc.)
    Get {
        surface: String,
        /// Property name: url, title, text, html, value, attr, count, box, styles
        property: String,
        /// Optional selector or element ref
        target: Option<String>,
        #[arg(long)]
        attr: Option<String>,
        #[arg(long)]
        property_name: Option<String>,
    },
    /// Navigate to a URL
    Goto {
        surface: String,
        url: String,
    },
    /// Go back
    Back { surface: String },
    /// Go forward
    Forward { surface: String },
    /// Reload
    Reload { surface: String },
    /// Evaluate JavaScript
    Eval {
        surface: String,
        expression: String,
    },
    // ... remaining actions
}
```

**Alternative (simpler, recommended):** Instead of enumerating every action in clap, use a hybrid approach. Define explicit subcommands for the most common actions (open, list, close, snapshot, click, fill, wait, get, goto) and a catch-all for less common ones. The catch-all forwards raw action + params to the socket generic proxy.

### Surface Handle Resolution Flow

```
CLI: cmux browser surface:7 click e1
  |
  v
CLI parses "surface:7" as target, "click" as action, "e1" as arg
  |
  v
CLI sends JSON-RPC: browser.click { "surface_ref": "surface:7", "target": "e1" }
  |
  v
Socket handler resolves surface:7 -> UUID via HashMap<u32, UUID>
  |
  v
If found: forward to BrowserManager with UUID
If not found: return { "error": "surface:7 not found", "available": [...] }
```

### JSON Default Output for Browser (D-03)

The browser subcommand inverts the default: JSON by default, `--no-json` for human-readable. This is a CLI-level concern, not a format module change. The `run()` function should check if the command is a `Browser` variant and flip the default.

```rust
// In run():
let json_mode = match &cli.command {
    Commands::Browser(_) => !cli.no_json,  // JSON default for browser
    _ => cli.json,                          // human-readable default for everything else
};
```

This requires adding a `--no-json` global flag alongside `--json`.

### Anti-Patterns to Avoid
- **Don't duplicate action parsing:** The socket already has a generic `BrowserAction` proxy. The CLI should map to JSON-RPC methods; don't create separate handler logic in the CLI.
- **Don't break backward compat silently:** Old flat commands (`cmux browser-open`, `cmux browser-click`) must remain functional (hidden in help). Test both paths.
- **Don't put ref resolution in the CLI:** The CLI sends `surface_ref: "surface:7"` to the socket. The app-side handler resolves it. The CLI should NOT maintain its own ref cache.

## Common Pitfalls

### Pitfall 1: Clap Subcommand Ordering Ambiguity
**What goes wrong:** If `surface:7` is both a valid subcommand name candidate and an argument, clap may misparse.
**Why it happens:** Clap tries to match subcommand names before positional args.
**How to avoid:** Make surface-targeted commands explicit subcommands where the surface ref is the first positional arg (e.g., `cmux browser snapshot surface:7 --interactive`), OR use `external_subcommand` to capture everything after `browser` as raw strings and parse manually.
**Warning signs:** `cmux browser surface:7 click e1` fails with "unknown subcommand: surface:7".

### Pitfall 2: Shell Completion Regeneration
**What goes wrong:** After restructuring the CLI enum, shell completions become stale.
**Why it happens:** Completions are pre-generated by `cmux-generate` binary, not at runtime.
**How to avoid:** Re-run `cargo run --bin cmux-generate` after CLI changes. Verify completions include new `browser` subcommands.
**Warning signs:** Tab completion shows old flat `browser-open` but not `browser open`.

### Pitfall 3: Backward Compat Flag Conflicts
**What goes wrong:** Old `--json` flag conflicts with new `--no-json` browser default.
**Why it happens:** Global `--json` flag applies to all subcommands including `Browser`.
**How to avoid:** Keep `--json` global. For browser commands, JSON is the default output. `--json` is effectively a no-op for browser commands (already JSON). Only `--no-json` changes behavior. Do NOT remove or redefine `--json`.
**Warning signs:** `cmux --json browser open https://example.com` behaves differently than `cmux browser open https://example.com`.

### Pitfall 4: RPM %files Glob Gotcha
**What goes wrong:** RPM `%files` with directory globs can either miss files or include the parent directory itself.
**Why it happens:** `%{_datadir}/cmux/skills/cmux/` means "include this directory and everything in it" but only if rpmbuild knows about it.
**How to avoid:** Use `%dir %{_datadir}/cmux/` for directory ownership, then list subdirectories. Or use `%{_datadir}/cmux/` to capture everything under it recursively.
**Warning signs:** RPM build succeeds but `rpm -ql cmux` doesn't show skill files.

### Pitfall 5: CMUX_WORKSPACE_ID Not Set
**What goes wrong:** `browser open` tries to read `CMUX_WORKSPACE_ID` from environment but it's not set for the terminal pane.
**Why it happens:** cmux-app must inject this env var into spawned terminal processes. If not already done, it won't be present.
**How to avoid:** Check whether cmux-app already sets `CMUX_WORKSPACE_ID` in the terminal spawn path. If not, this is a prerequisite task. The `--workspace` flag is the fallback.
**Warning signs:** `browser open` always creates in the first/active workspace instead of the caller's workspace.

### Surface Ref Parsing

```rust
/// Parse a surface reference like "surface:7" or a UUID string.
fn parse_surface_ref(input: &str) -> SurfaceTarget {
    if let Some(n) = input.strip_prefix("surface:") {
        if let Ok(id) = n.parse::<u32>() {
            return SurfaceTarget::ShortRef(id);
        }
    }
    if let Ok(uuid) = uuid::Uuid::parse_str(input) {
        return SurfaceTarget::Uuid(uuid);
    }
    SurfaceTarget::Invalid(input.to_string())
}

enum SurfaceTarget {
    ShortRef(u32),
    Uuid(uuid::Uuid),
    Invalid(String),
}
```

## Open Questions

1. **CMUX_WORKSPACE_ID injection**
   - What we know: The skill docs reference `CMUX_WORKSPACE_ID` for workspace-aware `browser open`. The codebase has `workspace_id` in SSH tunnel code but no evidence of injecting it as an env var into spawned terminals.
   - What's unclear: Whether cmux-app already sets this env var when spawning terminal surfaces.
   - Recommendation: Check the terminal spawn path in cmux-app. If not set, add it as a prerequisite task. The `--workspace` flag serves as fallback regardless.

2. **Wait command polling mechanism**
   - What we know: D-11 requires full wait support. The socket already proxies to agent-browser via `BrowserAction`.
   - What's unclear: Whether agent-browser's wait commands are blocking (return when condition met) or require polling from cmux side.
   - Recommendation: Test agent-browser wait behavior. If it blocks until timeout or condition, the CLI just needs a longer socket timeout for wait commands. If it returns immediately, cmux needs a poll loop.
