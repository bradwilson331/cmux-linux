# Phase 12.1: CLI Browser Parity & Skill Packaging - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

AI agents running inside cmux can discover and use browser automation skills, with a CLI syntax that matches the skill documentation (`cmux browser <surface> <action>`) and skills installed to a well-known path in .deb/.rpm packages. This phase bridges the gap between the existing flat browser CLI commands and the nested `cmux browser` syntax the skills document, adds surface handle management, and packages skills for agent discoverability.

</domain>

<decisions>
## Implementation Decisions

### CLI Syntax Restructure
- **D-01:** Nested subcommand approach — add a `Browser` variant to the `Commands` enum with its own sub-enum for actions (`open`, `click`, `fill`, `snapshot`, `wait`, etc.). Old flat `BrowserOpen`/`BrowserClick`/etc. become hidden aliases for backward compat.
- **D-02:** `cmux browser` requires a running cmux-app instance. All browser commands route through the socket to cmux-app, which manages agent-browser lifecycle. No standalone mode.
- **D-03:** Browser subcommand defaults to JSON output (agents are the primary consumers). Human-friendly output available via `--no-json` flag. This is opposite to the rest of the CLI which defaults to human-readable.
- **D-04:** Surface targeting via positional argument with `surface:N` prefix pattern. First arg after `browser` is parsed as surface target if it matches `surface:N` or UUID pattern. Otherwise treated as action name (e.g. `browser open`, `browser list`).

### Surface Handle Design
- **D-05:** Both short refs (`surface:N`) and full UUIDs accepted as input. Short refs mapped to UUIDs internally via a registry in cmux-app memory.
- **D-06:** Short ref registry lives in cmux-app memory as `HashMap<u32, UUID>`. Counter increments per `browser open`. Lost on restart (acceptable — browser panes don't survive restart either). CLI queries socket to resolve refs.
- **D-07:** `browser open` creates browser pane in the caller's workspace via `CMUX_WORKSPACE_ID` environment variable. `--workspace` flag overrides.
- **D-08:** Stale handle returns JSON error with available surfaces: `{"error": "surface:7 not found", "available": ["surface:3", "surface:5"]}`.
- **D-09:** `cmux browser list` returns table with ref, UUID, URL, and status. JSON mode returns array of surface objects.
- **D-10:** `browser open` routes through socket JSON-RPC `browser.open` command to cmux-app, which delegates to BrowserManager.
- **D-11:** Full `wait` support: `--selector`, `--text`, `--url-contains`, `--load-state`, `--function`, `--timeout-ms`. Essential for reliable automation flows.
- **D-12:** `snapshot --interactive` returns identical accessibility tree format as agent-browser (same element refs e1, e2, same tree structure).

### Skill Install & Discovery
- **D-13:** Package only `cmux` and `cmux-browser` skills in .deb/.rpm. Other skills (cmux-debug-windows, cmux-markdown, release) are repo-only — not user-facing for Linux packages.
- **D-14:** Static CLAUDE.md baked into the package at `/usr/share/cmux/CLAUDE.md`. References skill paths and summarizes socket API. Updated when package is upgraded.
- **D-15:** Skills installed to `/usr/share/cmux/skills/cmux/` and `/usr/share/cmux/skills/cmux-browser/`.
- **D-16:** Do NOT install agent-browser skills — cmux-browser skill already wraps that interface. Avoids confusing agents with duplicate interfaces.

### Package Integration
- **D-17:** Skills and CLAUDE.md added to existing cmux .deb and .rpm packages (no separate cmux-skills package). Small text files, no reason to split.
- **D-18:** CLAUDE.md content: skill directory paths, key socket commands, cmux CLI availability summary. Enough for Claude Code to discover capabilities without inlining full skill content.

### Claude's Discretion
- Exact clap enum structure and attribute macros for nested browser subcommand
- Short ref counter implementation details (AtomicU32 vs mutex-guarded)
- How `wait` commands poll agent-browser (WebSocket subscription vs periodic check)
- CLAUDE.md exact prose and formatting
- Whether to update packaging/scripts/build-deb.sh and build-rpm.sh inline or add a separate install-skills.sh step

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `BrowserManager` in `src/browser.rs` — already manages agent-browser lifecycle, socket discovery, auto-start, CDP streaming
- `which_agent_browser()` — binary discovery function with PATH and /usr/lib/cmux/ fallback
- Existing flat browser commands in `src/cli/mod.rs` — `BrowserOpen`, `BrowserClick`, `BrowserSnapshot`, etc. (25+ commands)
- Socket dispatch in `src/cli/mod.rs` — maps CLI commands to JSON-RPC method names
- `skills/cmux-browser/` — complete skill with SKILL.md, command reference, templates, agent configs

### Established Patterns
- clap `#[derive(Parser, Subcommand)]` for CLI structure
- JSON-RPC dispatch: `Commands::Variant => ("method.name", json!(params))` pattern
- Socket client sends request, receives response, formats based on `--json` flag
- `packaging/scripts/` bash scripts with `set -euo pipefail` and `REPO_ROOT` pattern

### Integration Points
- Socket command handlers in cmux-app (need new handlers for browser.open, browser.snapshot, etc. with surface ref support)
- `packaging/scripts/build-deb.sh` and `build-rpm.sh` — need to copy skills/ and CLAUDE.md
- `packaging/rpm/cmux.spec` — need to add %files entries for skills and CLAUDE.md

</code_context>
